Bitok Documentation

Bitcoin v0.3.19 with CPU-optimized proof-of-work

What is Bitok

Bitok is Bitcoin v0.3.19 (Satoshi Nakamoto's final release from December 2010) adapted to run on modern systems with GPU-resistant proof-of-work.

This is the last version of Bitcoin that Satoshi worked on, running as its own network with one critical change: a mining algorithm that keeps CPU mining viable.

Key Changes from Original Bitcoin

  1. Modern System Compatibility: Updated build system for OpenSSL 3.x, Boost 1.74+, GCC 11+, wxWidgets 3.2
  2. Yespower Proof-of-Work: Replaced SHA-256 with Yespower 1.0 for CPU-friendly, GPU/ASIC-resistant mining
  3. New Genesis Block: Separate network with fresh blockchain

Everything else is identical to Bitcoin v0.3.19: same transaction format, script system, networking code, wallet behavior, 21M cap, halving schedule. All security fixes from the Satoshi era are included from launch.

Why Bitcoin v0.3.19?

This version includes all critical security fixes (value overflow protection, checkpoints, DoS limits, IsStandard filtering) while maintaining the original Bitcoin design by Satoshi and his philosophy, before later protocol changes.

GPU Resistance:

With SHA-256, GPUs are 300x faster than CPUs. With Yespower, GPUs are only 2-3x faster and consume far more power, making them uneconomical for mining.

BitokPOW - Yespower Algorithm

Yespower is a CPU-optimized, memory-hard hashing algorithm designed to resist GPU and ASIC acceleration while remaining efficient on consumer CPUs.

Why Yespower?

On December 12, 2010, Satoshi Nakamoto wrote:

"We should have a gentleman's agreement to postpone the GPU arms race as long as we can for the good of the network. It's nice how anyone with just a CPU can compete fairly equally right now."

Bitcoin failed this requirement. SHA-256 was quickly dominated by GPUs, then ASICs, then industrial mining farms. Bitok enforces this requirement with Yespower.

Design Goals

Goal SHA-256 Yespower
CPU-friendly No Yes
GPU-resistant No Yes
ASIC-resistant No Yes
Laptop mining viable No Yes
Memory-hard No Yes (~128KB/hash)

Parameters

yespower_params_t {
    .version = YESPOWER_1_0,
    .N = 2048,              // Memory cost parameter (~128KB)
    .r = 32,                // Block size parameter
    .pers = "BitokPoW",     // Personalization string
    .perslen = 8
}

Key Features

  • Sequential Memory Initialization: Must fill 128KB before starting, cannot be parallelized
  • Random Memory Access: Pseudo-random reads create memory latency bottleneck
  • SIMD Optimizations: Automatically uses SSE2/AVX/AVX2/AVX512 (CPU advantage)
  • Domain Separation: "BitokPoW" personalization prevents cross-chain hash reuse

Why GPUs Can't Win

  • Memory Bandwidth: GPUs have high bandwidth but poor latency for random access patterns
  • SIMD Instructions: x86 CPUs have AVX2/AVX512 which Yespower exploits; GPUs have different SIMD that doesn't map efficiently
  • Power Efficiency: CPU uses 50-100W for reasonable hashrate; GPU uses 250-350W for only 2-3x improvement

Manifesto

I took Bitcoin v0.3.19, changed the mining algorithm so GPUs can't dominate, gave it a new genesis block, and ran it. That's the manifesto.

What Happened to Bitcoin

Bitcoin worked in 2010. You downloaded the software, clicked "Generate Coins," and your laptop found blocks. The network grew because participation was free and easy.

Then GPUs happened. Then mining pools. Then FPGAs, ASICs. The barrier to entry went from "own a computer" to "have access to cheap electricity and specialized hardware."

What Satoshi Said

December 12, 2010:

"We should have a gentleman's agreement to postpone the GPU arms race as long as we can for the good of the network."

To Laszlo Hanyecz (the pizza guy):

"GPUs are much less evenly distributed, so the generated coins only go towards rewarding 20% of the people for joining the network instead of 100%."

And:

"It's inevitable that GPU compute clusters will eventually hog all the generated coins, but I don't want to hasten that day."

What Bitok Changes

  1. Yespower instead of SHA-256: GPU miners don't get 1000x advantage anymore. More like 2x, if that.
  2. New genesis block: Separate network. No confusion with BTC. Clean slate.
  3. Modern build system: Compiles on Ubuntu 24.04, macOS, Windows.

What Bitok Does NOT Change

  • No new opcodes
  • No SegWit
  • No block size increase
  • No layer 2
  • No BIPs
  • No governance
  • No foundation
  • No roadmap

The protocol is frozen at December 2010.

RPC API

Bitok exposes a JSON-RPC 1.0 interface for programmatic access. This is essential for exchanges, mining pools, block explorers, and integrations.

Connection

# Start daemon with RPC
./bitokd -rpcuser=myuser -rpcpassword=mypass -server

# Protocol: JSON-RPC 1.0
# Transport: HTTP POST
# Port: 8332
# Binding: 127.0.0.1 (localhost only)

Python Example

import requests
import json

class BitokRPC:
    def __init__(self, user, password, host='127.0.0.1', port=8332):
        self.url = f'http://{host}:{port}/'
        self.auth = (user, password)
        self.headers = {'content-type': 'application/json'}
        self.id = 0

    def call(self, method, params=[]):
        self.id += 1
        payload = {
            'jsonrpc': '1.0',
            'id': self.id,
            'method': method,
            'params': params
        }
        response = requests.post(
            self.url,
            data=json.dumps(payload),
            headers=self.headers,
            auth=self.auth
        )
        return response.json()['result']

# Usage
rpc = BitokRPC('myuser', 'mypass')
info = rpc.call('getinfo')
print(f"Balance: {info['balance']} BITOK")
print(f"Blocks: {info['blocks']}")

Exchange Integration

Integrating Bitok into an exchange requires implementing deposit detection and withdrawal processing.

Deposit Address Generation

def generate_deposit_address(rpc, user_id):
    """Generate unique deposit address for user"""
    label = f"deposit_user_{user_id}"
    address = rpc.call('getnewaddress', [label])

    # Store in database
    db.execute(
        "INSERT INTO deposit_addresses (user_id, address, label) "
        "VALUES (?, ?, ?)",
        (user_id, address, label)
    )

    return address

Deposit Detection

def check_deposits(rpc, address, min_confirmations=6):
    """Check for confirmed deposits"""
    current_amount = rpc.call('getreceivedbyaddress', [address, 1])
    confirmed_amount = rpc.call('getreceivedbyaddress', [address, min_confirmations])

    if confirmed_amount > 0:
        # Credit user account
        credit_user(address, confirmed_amount)

Withdrawal Processing

def process_withdrawal(rpc, address, amount):
    """Process withdrawal request"""
    # Validate address
    valid = rpc.call('validateaddress', [address])
    if not valid['isvalid']:
        return {'error': 'Invalid address'}

    # Send transaction
    txid = rpc.call('sendtoaddress', [address, amount])
    return {'success': True, 'txid': txid}

Security

Code Security

This is 2010 code adapted for 2024. The cryptography is fine (ECDSA, SHA-256 for non-mining hashes). All security fixes from Satoshi's final release are included:

  • Value overflow protection (184B coin bug)
  • Blockchain checkpoints
  • DoS limits
  • IsStandard() filtering

Wallet Security

Critical: Back up your wallet.dat file. If you lose it, coins are gone forever. There is no recovery mechanism.

Backup Locations

  • Linux: ~/.bitok/wallet.dat
  • macOS: ~/Library/Application Support/Bitok/wallet.dat
  • Windows: %APPDATA%\Bitok\wallet.dat

RPC Security

Best practices for securing the RPC interface:

  • Strong credentials: Use 32+ character random passwords
  • Firewall rules: Restrict RPC port (8332) to trusted IPs only
  • Local binding: RPC binds to 127.0.0.1 by default - keep it that way
  • SSH tunneling: For remote access, use SSH tunnels instead of exposing the port

Network Security

Bitok uses IRC bootstrap for peer discovery (same as original Bitcoin). Connects to irc.libera.chat channel #findsatoshi.
